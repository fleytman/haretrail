# Doc Write workflow

## Канонические пути

`{data-repo}` — root приватного HARE Trail data repo. Resolve it through `HARETRAIL_DATA_DIR`, then the current workspace if it has the expected data shape, then an explicit user/host-tool path. Do not hardcode personal absolute paths.

- Точка входа: `{data-repo}/AGENTS.md`
- Базовые правила: `{data-repo}/BASE.md`
- Work artifacts: `{data-repo}/work-artifacts/`
- Durable notes: `{data-repo}/notes/`

## Цель

Сделать документацию:

- короткой;
- понятной человеку;
- согласованной с правилами репозитория;
- без длинных code dumps;
- достаточно конкретной, чтобы ей можно было пользоваться.

## 1. Сначала прочитать стиль целевого места

Перед написанием:

1. Найти `AGENTS.md`, `README`, `docs/`, шаблоны и соседние документы.
2. Определить язык существующей документации.
3. Понять аудиторию:
   - команда проекта;
   - пользователь локальных notes;
   - сам автор как будущий читатель.

## 2. Язык по умолчанию

- Для рабочих инженерных репозиториев по умолчанию писать по-английски, если пользователь не попросил другое.
- Для личных заметок, home tasks и data-repo artifacts по умолчанию можно писать по-русски.

## 3. Что писать

Док должен отвечать на реальные вопросы читателя:

- что это;
- зачем нужно;
- когда использовать;
- как запустить или применить;
- какие ограничения и важные нюансы.

## 4. Чего избегать

- длинных вставок кода без необходимости;
- пересказа diff по файлам;
- лишней теории, если пользователю нужна рабочая инструкция;
- внутреннего монолога ИИ;
- дублирования уже существующей канонической doc.

## 5. Источники для доки

Использовать:

- код и соседние docs в репозитории;
- relevant work-artifacts;
- summaries и prompts, если дока выросла из исследования;
- confirmed facts из tracker/journal, а не сырые догадки.

## 6. Если документ рождается из рабочей заметки

Частый паттерн:

- сырая заметка в `work-artifacts/`
- потом clean version в repo docs

В этом случае:

- не копировать рабочий лог в документацию;
- вытащить только устойчивое знание;
- если есть важные caveats, оставить их кратко и по делу.
