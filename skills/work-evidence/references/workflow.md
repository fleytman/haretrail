# Work Evidence workflow

Движок сбора evidence о работе за период. Выдаёт нормализованный source-bound ledger для
потребителей (`daily`, `contribution-log`). Сам ничего не «рендерит для человека».

## Канонические пути

`{data-repo}` — root приватного HARE Trail data repo (resolve через `HARETRAIL_DATA_DIR`,
затем текущий workspace, затем явный путь).

- Рабочие заметки/дебрифы: `{data-repo}/work-artifacts/`, `{data-repo}/session-debriefs/`,
  `{data-repo}/LESSONS.md`.
- Потребитель сам выбирает, куда писать результат (`daily/`, contribution-log container и т.п.)
  и складывает сырьё источников в свой `sources/`.

## Конфиг (`~/.haretrail/`)

Источники и параметры НЕ хардкодим — берём из конфига; если его нет, спрашиваем у пользователя
и сохраняем. Ключи:

- `HARETRAIL_TRACKER` — трекер задач (напр. `jira`) + как ходить (MCP/CLI/API).
- `HARETRAIL_CHAT` — чат (напр. `slack`) + способ доступа.
- `HARETRAIL_CODEHOST` — code-host (напр. `github`) + login/org.
- `HARETRAIL_PULL_STRATEGY` — `all` | `partial` (тянуть всё или часть).
- `HARETRAIL_WORK_FILTER` — правила «работа vs личное» (org, тикет-префиксы, личные пути/репо).
- `HARETRAIL_LANG` — язык вывода.

Если трекер/чат не настроены или MCP недоступен — сказать пользователю и предложить `/mcp`
(не обходить через curl/bash).

## 1. Резолв периода

- Период приходит от потребителя (`daily` — короткий, `contribution-log` — длинный) или из запроса
  пользователя. «пн / с прошлого дейли / за неделю» → конкретные `YYYY-MM-DD`.
  `date '+%Y-%m-%d %A'`; начало недели — `date -v-mon` (macOS) / `date -d 'last monday'`.
- Если дата сессии сместилась — пересчитать и добрать «хвост».

## 2. Сбор по источникам (по агенту на источник, параллельно)

Сырой отчёт каждого источника сохранять в `<container>/sources/<source>.md`.

### 2.1 Code-host (напр. GitHub через `gh`)
Различать **тип участия** и ключевать по **времени действия** (НЕ по дате создания объекта):

| Тип | Запрос (пример для gh) | Время | Чьё |
|---|---|---|---|
| Создал | `search prs --author <login> --created S..E` | created | моё |
| Влил/закрыл | `search prs --author <login> --merged S..E` / `--closed` | merged/closed | моё |
| Коммитил (вкл. чужие) | `search commits --author <login> --author-date S..E` → sha→PR | commit date | моё/чужое |
| Ревьюил | `search prs --reviewed-by <login>` → reviews, фильтр `submitted_at` | review date | чужое (помощь) |
| Комментил | `search prs --commenter <login>` → фильтр дат, дедуп vs ревью | comment date | чужое |
| Ждут меня | `search prs --review-requested <login> --state open`, `--assignee` | — | pending |

Правила: `--created` НЕ прокси активности (объект, созданный до окна, но влитый/закоммиченный в
окне, ловится только `--merged`/`--closed`/`search commits`). Built-in date-qualifier ключует по
своему таймстемпу; для review/comment/commit нужен второй проход по их собственным датам. Статус —
из API объекта, не из нарратива. Дедуп (один объект в нескольких корзинах). Свои → «моя работа»,
ревью/комменты/коммиты в чужое → «помощь».
Известный риск: индекс приватных репо в `search commits` лагает (false negative) — фолбэк на
events API.

### 2.2 Трекер (напр. Jira)
- Запросы: `assignee = currentUser() AND updated >= START`, плюс `reporter`, плюс created;
  по каждому тикету — changelog.
- **ОБЯЗАТЕЛЬНО:** выписать путь по статусам в окне И **кто** выполнил каждый транзишен.
  Интерпретация (кто двигал = чьё действие):
  - лид/другой создал/вытащил из бэклога и назначил на меня → задачу выдал лид (входящая);
  - я создал и вытащил → сам себе назначил;
  - **лид сменил статус — НЕ всегда «лид откатил»**: может быть (а) реприоритизация (решение лида)
    или (б) фиксация на дейли/планинге статуса, который я забыл выставить (тогда это мой прогресс).
    Различать по реальной работе за транзишеном; неясное — пометить;
  - массовый Closed→Done одним человеком по куче тикетов → миграция воркфлоу, не завершение.
  **Кто нажал кнопку ≠ чьё решение** — не приписывать себе чужие решения и не отдавать другим свой прогресс.

### 2.3 Чат (напр. Slack)
- Сообщения `from:<user>` за окно + чтение ключевых тредов. Фиксировать КТО спросил/ответил/решил.
- Главный источник для invisible/coordination и для открытых вопросов/блокеров.

### 2.4 Рабочие заметки и дебрифы (data-repo)
- Датированные записи `work-artifacts/*/journal.md|tracker.md` за окно (читать ЦЕЛИКОМ, не грепом).
  Часто дают «скрытые» задачи, которых нет тикетами.
- **Чек уроков/дебрифов:** `LESSONS.md` + `session-debriefs/` за период — важные инсайты/уроки.
- Осторожно с migration-noise: доверять датированным записям, а не raw mtime.

## 3. Фильтр «личное vs работа»

- По `HARETRAIL_WORK_FILTER`: работа = настроенный org + тикет-префиксы; личные пути/репо/проекты
  исключать. Базовая эвристика, уточнения сохранять в конфиг.
- Неоднозначное — не тащить в рабочий вывод по умолчанию.

## 4. Дедуп и нормализация → evidence ledger

- Свести один и тот же пункт из разных источников в одну запись.
- Нормализованный work item: `{тип, тикет, ссылка, дата действия, статус, кто-действовал,
  кому-помог, источник(и), суть}`.
- Ledger отдаётся потребителю как есть; потребитель решает группировку/формат/секции.

## 5. Роллап

Для больших периодов потребитель (`retro`, `contribution-log`) может агрегировать уже готовые
дейли-записи (`daily/*/standup.md`) вместо повторного сбора с нуля; движок добирает только то, чего
нет в дейликах.

Латентность: движок — это **«медленный слой»** (минуты, время задаёт code-host-агент). Для быстрого
ответа потребители (`daily`/`retro`) сначала берут готовое (буфер, последние `sources/`, дейлики), а
живой прогон движка пускают фоном или по schedule.

## 6. Что НЕ делать

- Не выдавать «по памяти» без evidence; не рендерить человеко-facing отчёт (это работа потребителя).
- Не считать draft/open влитым; не включать действия вне окна.
- Не приписывать себе чужие транзишены/решения; не отдавать другим свой прогресс.
- Не тащить личное в рабочий вывод; не обходить недоступный MCP через curl/bash.
